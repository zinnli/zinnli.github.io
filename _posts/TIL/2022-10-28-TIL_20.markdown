1.   환장하는 깃허브로 협업하기!

     <br/>
     백엔드 팀에서 깃허브 organization으로 초대를 해주셨는데.. 폴더 구조를 제작한 파일이 제대로 업로드되지 않았다. 업로드가 안된다는 에러 코드를 진짜 과장 안하고 20번은 본 것 같다..
     <br/>
     허가가 안되어있다고 folk로 가져가라고 떠서 folk로 가져가서 작업했는데 그때도 폴더구조를 짜둔 상태를 깃에 올려놓고, 추가 작업을 한 뒤에 브랜치를 파서 올리려고 하니..... main에 있던 내용이 branch로 넘어오면서 작업했던 파일을 1차로 날려버렸다.
     <br/>

     그리고 이대로 안되겠다 싶어서 90%가량 복구(라고 쓰고 다시 쓰기라고 읽는다)하고 organization에 있는 레포와 연결되어있는지 확인해봤는데 분명 연결이 되어있었는데 어느 순간부터 folk도 끊겨버린 것을 확인해버렸다.. 예상하는 바로는 팀원분이 작업한 부분을 올리시다가 변경하신 것 같다.... 팀원분도 그런 변경사항을 모르셨던 것 같아서 그냥 몰랐던 척 했다..히히.. 너무 슬퍼..
     <br/>

     아무튼 해당 레포에 올리는 것을 3-5번정도 더 시도해봤는데 동일한 문제가 반복되어서 그냥 작업한 파일들을 따로 백업해두고 organization의 레포를 클론해서 가져왔다. 다음부턴 이렇게 시간을 태우기 보다는 깔끔하게 포기하고 clone하고 백업한 부분을 확인하고 추가하는 게 낫지않을까 싶다..
     <br/>

     이 과정에도 작업물을 한 번 더 날려서 진짜 너무 속상했다.

      <br/>
      이런 과정에서 사용했던 방법들

     ```react

     //연결된 레포 확인
     git remote -v

     // 연결되어있는 모든 레포가 뜸(각 저장소의 이름?이 앞에 뜨는 것을 확인할 수 있었음)
     //여기서 나같은 경우는 2개의 레포가 떴고(아마 folk때문인듯) organization에 있는 레포에는 푸쉬가 되지않고 folk해온 레포에만 푸쉬가 되었음

     //저장소 삭제(레포 연결 해제)
     git remote remove origin //origin이 보통 제일 기본 저장소

     //저장소 추가(레포 연결)
     git remote "레포 주소"

     //------------ 오류(이 부분을 종류별로 10번 이상 시도해봄,, 내시간 아깝꾸,,)


     //레포 클론해오기
     git clone "레포주소"

     //여기서부턴 평소와 동일하게 커밋
     //branch를 만들 땐 풀로 먼저 땡겨놓고(개인 작업 전에 꼭 풀 제발 먼저 브랜치 따기!!!!!!!!!!!!!!!!!!!!!!!!) 코딩 짜기..

     ```
